# WORKFLOW

このファイルは、経営課題をCodexで実行できる作業へ変える流れをまとめたものです。

## 基本フロー

```text
経営者との会話
  -> 課題の整理
  -> 最初の成果物を決める
  -> data/ に入力を置く
  -> Codexに小さく依頼する
  -> output/ で結果を見る
  -> WORKLOG.md に記録する
  -> TASKS.md に次の改善を残す
```

## Step 1: 会話から始める

最初に文書を書き切る必要はありません。

経営者が困っていること、使っているExcel、手作業の流れ、最初に見たい結果をCodexに話します。

Codexとの会話から、`BUSINESS.md`、`RULES.md`、`SERVERS.md`、`TASKS.md` の下書きを作ります。

## Step 2: 最小の成果物を決める

最初から完成システムを作りません。

良い例:

```text
売上CSVから月別売上と上位商品を出すMarkdownレポートを作る。
```

大きすぎる例:

```text
会社全体の売上管理システムを作る。
```

## Step 3: データを置く

最初は、`data/` にサンプルデータやコピーしたデータを置きます。

例:

- `data/sales.csv`
- `data/products.xlsx`
- `data/inquiries.csv`
- `data/urls.txt`

機密情報や個人情報がある場合は、サンプル化または匿名化します。

## Step 4: Codexに小さく依頼する

一度に大きく頼まず、1つの具体的な成果物を依頼します。

```text
BUSINESS.md、RULES.md、SERVERS.md、TASKS.mdを読んでください。
data/sales.csvを読み、月別売上と上位商品をoutput/sales_summary.mdに出すPythonスクリプトを作ってください。
実行方法と結果をWORKLOG.mdに追記してください。
```

## Step 5: 実行して確認する

作ったコードは必ず実行します。

確認するもの:

- コマンドが動くか
- `output/` に結果が出るか
- 経営者が結果を理解できるか
- エラーがあれば何が原因か

## Step 6: 記録する

`WORKLOG.md` に残すもの:

- 実行コマンド
- 変更したファイル
- 出力結果
- エラー
- 次に頼むこと

## Step 7: 次の改善へ進む

`TASKS.md` に、次の改善を小さく書きます。

```text
Next Improvements
- 前月比を追加する
- 上位商品だけでなく利益率も出す
- HTMLで見られるようにする
```

## 運用ワークフローを作るとき

投稿、動画生成、外部API連携、メール送信、定期処理など、時間がかかる処理は「1回動いたら終わり」ではなく、運用として設計します。

最初に決めること:

- 目標: 1日何件、何時まで、または無限ループで何を改善し続けるか
- 実行単位: 1ジョブで何を入力し、何を出力するか
- 重複防止: 同じ記事、同じURL、同じ投稿を二重生成しない条件
- 状態: queued / running / done / skipped / failed / canceled の意味
- 失敗理由: 「失敗」だけでなく、重複、入力不足、外部APIエラー、認証切れを分ける
- 再実行: どの失敗は再実行し、どの失敗はスキップ扱いにするか
- 証跡: 生成物URL、投稿URL、コピー文、実行ログ、スクリーンショットなど

AIに依頼するときは、単に「動かして」ではなく、次のように依頼します。

```text
この処理をジョブとして設計してください。
1日の目標件数、重複防止条件、成功/スキップ/失敗のステータス、再実行条件をTASKS.mdに書いてください。
実装後はWORKLOG.mdに、実行コマンド、ジョブID、出力URL、失敗時の表示内容を残してください。
```

## テストは「見た」ではなく証跡を残す

Web UIや投稿機能では、コードを読んだだけで完了にしません。

確認例:

- ボタンを実際に押し、コピーされた文字列を `WORKLOG.md` に貼る
- 投稿APIのレスポンスIDと公開URLを書く
- 画面の表示文言が利用者向けとして自然か確認する
- 管理者向け文言が一般公開画面に出ていないか確認する
- 「本日完了」などの日付・時刻ステータスが実データと一致するか確認する

## この流れの意味

VWorkは、経営者の頭の中にある課題を、AIが実行できる形へ変えていく作業です。

そのために、会話、文書、コード、実行結果、次の改善を1つのフォルダに残します。
