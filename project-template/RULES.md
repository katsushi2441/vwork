# RULES

このファイルは、Codex/Claudeがこの案件で守る作業ルールです。

依頼者が毎回細かく指示しなくても、AI側がここを読んで安全に作業できる状態を作ります。

## 基本姿勢

- 作業前に `WORK_PROTOCOL.md` と `AGENTS.md` を読む
- 最初から大きな完成システムを作らない
- 1日以内に確認できる小さな成果物から始める
- 既存ファイル、既存データ、既存運用を確認してから作業する
- 人が編集した内容を勝手に戻さない
- 作業したら、変更内容・確認方法・未確認事項を報告する
- 動いていないもの、確認していない公開URL、偽物の画面や動画を「完了」として扱わない

## データ変更前の確認

次の処理は、実行前に必ず依頼者へ確認します。

- 既存データの上書き
- UPDATE / DELETE / upsert
- 大量データの一括変更
- 本番環境への反映
- 外部サービスへの投稿・送信
- メール送信、SNS投稿、決済、顧客連絡

## ジョブ・自動化のルール

定期実行、常駐サービス、キュー、投稿、動画生成などを作る場合は、実装前に次を `TASKS.md` または `SERVERS.md` に書きます。

- 1日の目標件数
- 1回の実行件数
- 実行間隔
- 重複防止条件
- 成功、スキップ、失敗、キャンセルの定義
- 再実行してよい条件
- 管理画面やログに表示するメッセージ

同じ記事、同じURL、同じ投稿、同じ動画を二重生成しないことを優先します。

外部APIエラー、認証切れ、入力不足、重複スキップは、同じ「失敗」にまとめず、理由が分かるように記録します。

## 秘密情報の扱い

以下はMarkdownやGit管理ファイルに書きません。

- APIキー
- パスワード
- FTP情報
- OAuthトークン
- 顧客の個人情報
- 本番データベース接続情報

必要な場合は `.env` やローカル環境変数を使い、場所だけを `SERVERS.md` に書きます。

設定が増えたら、値の入っていない `.env.sample` や `config.yml.sample` を作り、何を設定すべきかだけをGit管理します。

## HTML・Webページを作る時

HTMLを作る場合は、依頼されていなくても次を確認します。

- title / description / canonical / OGP画像
- Google Analyticsやsimpletrack.phpなど既存サイトの計測タグ
- PCとスマホでの表示崩れ
- 印刷・PDF化用の `@media print`
- HyperFramesで動画化しやすい見出しとセクション構成
- 公開URL、リンク、画像、OGPの反映

## HP・LP・UI制作ではDESIGN.mdを作る

ホームページ、ランディングページ、サービス紹介ページ、UI、営業資料など、見た目の品質が成果物の価値に直結する案件では、実装前に `DESIGN.md` を作ります。

`DESIGN.md` がない場合、Codex/Claudeは勝手に作り始めず、次を確認してから作業します。

- 対象ユーザー
- ページの目的
- 会社や商品の印象
- ファーストビューで伝えること
- CTA
- 参考にする既存ページ
- 避けたい見た目や表現
- 色、フォント、余白、写真、画像の方針
- スマホ表示で重視すること

`DESIGN.md` は全案件で必須ではありません。Excel集計やCSV処理など、見た目が主価値ではない案件では作らなくて構いません。

## 作業後に必ず残すこと

作業後は `WORKLOG.md` に次を残します。

- 何を依頼されたか
- 変更したファイル
- 実行したコマンド
- 出力されたファイル
- 確認したこと
- エラーや未確認事項
- 次にCodexへ頼むとよいこと

UIや投稿機能では、可能な限り証跡も残します。

- 実際にコピーされた文字列
- 投稿APIのレスポンス
- 公開URL
- 画面に表示されたステータス
- 失敗時に利用者へ見えるメッセージ
